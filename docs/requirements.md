# 1. Введение

## 1.1 Цель

Данная спецификация требований описывает функциональные и нефункциональные требования к веб-платформе онлайн-бронирования и продажи дополнительных услуг базы отдыха The Lagom версии 2.0.
Документ предназначен для команды разработки, тестирования и других участников проекта, задействованных в реализации и сопровождении системы.
Если не указано иное, все требования, описанные в документе, относятся к версии 2.0 и имеют высокий приоритет.

## 1.2 Глоссарий

В документе используются следующие обозначения:

- **FR** — функциональное требование;
- **NFR** — нефункциональное требование;
- **MVP** — минимально жизнеспособная версия продукта;
- **Пользователь** — гость базы отдыха, использующий систему бронирования;
- **Администратор** — сотрудник базы отдыха, управляющий бронированиями и услугами.

Термин «система» используется для обозначения веб-платформы онлайн-бронирования The Lagom.

# 2. Бизнес-требования

## 2.1 Организация

Организацией-владельцем системы является база отдыха The Lagom, предоставляющая услуги краткосрочного проживания и дополнительных сервисов (размещение, питание, SPA, досуг).

Цель организации — предоставление гостям комфортного загородного отдыха с возможностью онлайн-бронирования и приобретения дополнительных услуг через веб-платформу.

Система разрабатывается как цифровой канал взаимодействия с клиентами и инструмент увеличения выручки за счёт онлайн-продаж услуг и повышения конверсии бронирования.

## 2.2 Бизнес-задачи

В рамках разработки веб-платформы онлайн-бронирования и дополнительных услуг базы отдыха The Lagom выделяются следующие бизнес-задачи:

| # | Задача | Цель |
|----------|----------|----------|
| BT-1 | Увеличение выручки | Обеспечить рост среднего чека за счёт внедрения продажи дополнительных услуг (SPA, питание, досуг, ранний заезд, поздний выезд и др.) в процесс бронирования |
| BT-2 | Повышение конверсии бронирования | Сократить количество незавершённых бронирований за счёт: предоставления полной информации об услугах, упрощения процесса выбора и оформления брони, внедрения понятного пользовательского сценария |
| BT-3 | Автоматизация процессов обслуживания клиентов | Снизить нагрузку на администраторов за счёт автоматизации: обработки заявок на бронирование, подтверждения доступности услуг, управления дополнительными сервисами |
| BT-4 | Повышение качества пользовательского опыта | Обеспечить удобный и прозрачный процесс взаимодействия пользователя с системой: понятная структура услуг, единый процесс бронирования, минимизация необходимости ручных уточнений |
| BT-5 | Получение аналитических данных | Обеспечить сбор данных о: поведении пользователей, популярных услугах, конверсии бронирования, структуре доходов |

## 2.5 Операционная среда

Система представляет собой веб-платформу онлайн-бронирования, доступную пользователям через сеть Интернет.

Доступ к системе осуществляется через современные веб-браузеры, включая:
- Google Chrome;
- Safari;
- Mozilla Firefox;
- Microsoft Edge.

Система должна корректно функционировать на:
- персональных компьютерах;
- планшетах;
- мобильных устройствах.

Для обработки отдельных бизнес-процессов система взаимодействует с внешними сервисами:
- платёжной системой;
- сервисом отправки уведомлений (email/SMS).

Для работы системы требуется стабильное интернет-соединение.

## 2.6 Ограничения, предположения, зависимости

### Ограничения

- Система разрабатывается как веб-платформа и не предусматривает отдельного мобильного приложения в рамках версии MVP.
- Интеграция с внешними сервисами осуществляется через доступные API сторонних систем.
- Поддержка устаревших браузеров (Internet Explorer и аналогичных) не предусматривается.
- В рамках MVP не реализуется программа лояльности и динамическое ценообразование.

---

### Предположения

- Пользователь имеет стабильное подключение к сети Интернет.
- Пользователь использует современный веб-браузер.
- Внешние сервисы оплаты и уведомлений доступны и функционируют корректно.
- Администраторы системы обладают базовыми навыками работы с веб-интерфейсами.

---

### Зависимости

- Работа системы зависит от доступности платёжной системы.
- Отправка уведомлений зависит от работы внешнего email/SMS сервиса.
- Корректность отображения информации зависит от актуальности данных, вводимых администраторами.

| Область действия | Веб-сервис The Lagom |

### Базовый сценарий

1. Пользователь инициирует создание бронирования  
2. Система отображает календарь доступных дат  
3. Пользователь выбирает даты заезда и выезда  
4. Система проверяет доступность размещения  
5. Система отображает доступные варианты размещения  
6. Пользователь выбирает вариант размещения  
7. Система рассчитывает стоимость бронирования  
8. Пользователь подтверждает создание бронирования  
9. Система сохраняет бронирование и отображает его данные пользователю  

---

### Расширения

3a. Пользователь выбирает дополнительные услуги  
- Система отображает список доступных дополнительных услуг  
- Пользователь выбирает дополнительные услуги (завтрак, сауна, аренда велосипеда и т.д.)  
- Система добавляет стоимость услуг в итоговую стоимость бронирования  
- Пользователь возвращается к основному сценарию выбора размещения  

4a. Выбранные даты недоступны  
- Система отображает сообщение об отсутствии доступности  
- Пользователь возвращается к выбору дат  

6a. Пользователь не выбрал размещение  
- Система не позволяет продолжить создание бронирования  

---

### Изменения в технологиях и данных

- Создаётся запись бронирования в системе  
- Рассчитывается итоговая стоимость проживания

---

### Базовый сценарий

1. Пользователь открывает данные созданного бронирования  
2. Система отображает информацию о бронировании и итоговую стоимость  
3. Пользователь выбирает действие “Оплатить бронирование”  
4. Система перенаправляет пользователя к платёжному сервису  
5. Пользователь вводит платёжные данные и подтверждает оплату  
6. Платёжный сервис подтверждает успешную оплату  
7. Система обновляет статус бронирования  
8. Система отображает подтверждение успешной оплаты  

---

### Расширения

5a. Платёж отклонён  
- Система получает отказ от платёжного сервиса  
- Пользователь информируется о неуспешной оплате  
- Статус бронирования не изменяется  

4a. Платёжный сервис недоступен  
- Система отображает сообщение об ошибке  
- Пользователь может повторить попытку позже  

---

### Изменения в технологиях и данных

- Обновляется статус бронирования 
- Фиксируется транзакция оплаты

2. Система отображает список заявок

4. Система отображает подробную информацию о бронировании  
5. Администратор проверяет доступность размещения  
6. Администратор подтверждает бронирование
